# LLM Wiki パターン リファレンス

> **LLMが永続的なWikiをインクリメンタルに構築・維持する**
> カテゴリ: 手法・パターン *(references)*
> 最終更新: 2026-05-30

## 概要

**LLM Wiki** は Andrej Karpathy が提携した、LLM を活用した個人向け知識ベース構築パターン。

RAG（質問時に都度チャンクを検索して回答生成）とは根本的に異なる。
Wiki を **永続的な蓄積物（compounding artifact）** として、
情報源を追加するたびに LLM が Wiki の各所を更新していく。

## RAG との違い

| RAG | LLM Wiki |
|-----|----------|
| 質問時に都度チャンクを検索 | Wiki は永続的に蓄積 |
| 毎回ゼロから知識を再発見 | クロスリファレンス・矛盾チェック済み |
| 5文書の統合成が必要な質問は毎回苦労 | 統合成がもう Wiki に入っている |
| NotebookLM, ChatGPT ファイルアップロード型 | 知識の「コードベース」型 |

## コンセプト

Wikiは **一度コンパイルされ、その後継続的に最新に保たれる**。

- クロスリファレンスはもう Wiki 内に存在する
- 矛盾はすでにフラグ付け済み
- 読んだものはすべて Wiki に反映済み
- Wiki は読むたび・質問するたびに豊かになる

人間の役割は素材の収集・探索・正しい質問をすること。
LLM の役割はそれ以外のすべて（要約・相互参照・整理・記録管理）。

実運用イメージ：「LLM エージェントを片側に開き、Obsidian をもう片側に開く。
LLM が対話に基づいて編集し、リアルタイムで結果をブラウズする」。
Obsidian が IDE、LLM がプログラマー、Wiki がコードベース。

## 適用例

- **Personal**: 目標・健康・心理学・自己改善。日記、記事、ポッドキャットメモから自己の構造化された像を構築
- **Research**: 研究テーマを数週間〜数ヶ月かけて深掘り。論文・記事・レポートを読みながら包括的なWikiを構築
- **読書**:
 各章を読みながらキャラクター・テーマ・プロットのWikiページを構築。
 Gateway(Tolkien) 的なファンWiki を個人で作れる
- **Business/team**: Slackスレッド・会議書き起こし・プロジェクト文書・顧客通話から社内Wikiを構築
- **競合分析、デューデリジェンス、旅行計画、授業メモ、趣味の深掘り** 等、あらゆる

## アーキテクチャ（3層構造）

### 1. Raw Sources

情報源の策展されたコレクション。記事、論文、画像、データファイル等。
変更しない（イミュータブル）。これが信頼の源（source of truth）。

### 2. The Wiki

LLMが生成したMarkdownファイル群のディレクトリ。
要約、エンティティページ、コンセプトページ、比較表、概要、統合など。
LLMがこの層を完全に保有する：

- 新しいソースが入ってきた時にページ作成
- 既存ページの更新
- クロスリファレンスの維持
- 一貫性の保持

人間は読むだけ。LLM が書く。

### 3. The Schema

Wiki の構造、コンベンション、ワークフローを指示するドキュメント
（CLAUDE.md や AGENTS.md 等）。
LLM を「一般的なチャットボット」ではなく「規律あるWikiメンテナー」にするための
重要な設定ファイル。使うドメインに合わせて人間とLLMがともに育てていく。

## 基本操作

### Ingest（取り込み）

RawコレクションにソースをドロップしてLLMに処理させるフロー：

1. LLM がソースを読み、要点の対話をする
2. Wiki サマリーページの作成
3. インデックス更新
4. Wiki全体の関連エンティティ・コンセプトページの更新
5. ログ追記

1つのソースが10〜15ページに影響を与えることもある。
Karpathy本人は1つずつ処理して関与することを推奨（要約を読んでフィードバックできる）。

### Query（クエリ）

Wikiに質問する。LLM は関連ページを検索し、引用付きで回答を合成する。

回答は問いに応じて様々な形式を取る：

- Markdownページ（共通）
- 比較テーブル
- スライドデッキ（Marp）
- チャート（matplotlib）
- Canvas

重要な洞察： **質問の良い回答もWikiに新しいページとして保存できる。**
比較表、分析、発見されたつながりは対話履歴に消していいものではない。
このように探索もソースと同じように知識に蓄積されていく。

### Lint（定期健診）

定期的に Wiki のヘルスチェックをする。LLM に見つけさせる：

- ページ間の矛盾
- 新しいソースで古くなった記述
- リンクリストのない孤立ページ
- 言及されているがページのない重要コンセプト
- 欠落しているクロスリファレンス
- Web検索で埋められそうなギャップ

Wiki が成長するにしたがって健全性を保てる。

## インデックスとロギング

成長するWikiをナビゲートするための2つの特別ファイル：

### index.md（コンテンツ指向）

Wiki 内の全ページのカタログ。1行サマリーメタデータ付き。
カテゴリ別（エンティティ、コンセプト、ソース等）に整理。Ingest時に更新。
クエリに答える際、LLM はまずインデックスを読んで関連ページを見つけ、
そこに掘り下げる。
中規模（ソース100件、ページ数百）では十分機能し、
embeddingベースのRAGインフラが不要になる。

### log.md（時系列指向）

何がいつ起きたかの一追記専用記録。Ingest、Query、Lint を記録。
エントリが `## [YYYY-MM-DD] ingest | タイトル` のように一貫したプレフィックスで始まると、`grep` や `tail` で簡単にパースできる。

```bash
grep "^## \[" log.md | tail -5   # 最新5件
```

## ツール（任意）

Wikiが拡大するにつれて、LLM が効率的に操作するためのツールを自作できる。
検索エンジンが最も代表的なもの。小規模ならインデックスで十分だが、
成長するとプロパな検索が必要になる。

[qmd](https://github.com/tobi/qmd) が選択肢の1つ：BM25/ベクトルハイブリッド検索 + LLM リランキングの
ローカル検索エンジン。CLI にも MCP サーバーにも対応。

## Tips & Tricks

- **Obsidian Web Clipper**: Web記事をMarkdownに変換するブラウザ拡張。リップ収集に最適
- **画像をローカルDL**: Obsidian設定 → 添付ファイルパスを固定ディレクトリに →
  ホットキーにDLを割り当て → クリップ後に押せば全部ローカルディスクに落ちる
- **Obsidian グラフビュー**: Wiki の「形」を確認する最良の方法。ハブ、孤立、つながりが見える
- **Marp**: Markdown ベースのスライドデッキ。Obsidian プラグインとして利用可能
- **Dataview**: Wiki ページのYAML frontmatterに問い合わせるObsidianプラグイン。動的テーブル・リストを生成できる
- **git管理**: Wikiは単なるMarkdownファイル群のリポジトリ。バージョン管理・ブランチ・コラボが無料

## このパターンが機能する理由

知識ベース維持のつまらない部分は読むことや考えることではない。
それは記録管理（bookkeeping）だ。

- クロスリファレンスの更新
- 要約の最新維持
- 新データが古い主張と矛盾する際の指摘
- 数十ページにまたがる一貫性の維持

人間は維持管理の負担が価値を超えると Wiki を放棄するが、LLM は：

- 飽きない
- クロスリファレンスの更新を忘れない
- 1回で15ファイルに手を入れられる
- 維持コストがほぼゼロ

人間の仕事：キュレーション、分析の方向付け、正しい問いを立てること、意味を考えること。
LLMの仕事：それ以外のすべて。

このアイデアは Vannevar Bush の Memex（1945）に通じる。
個人的で積極的に策展された知識ストアと文書間の連想的道という彼のビジョンは、
Web の実態よりもこの Wiki パターンに近い。
維持を誰がするかという問題が解けなかったが、LLM がそれを担う。

## 注意

このドキュメントは意図的に抽象的。アイデアを伝えるものであり、特定の実装ではない。
ディレクトリ構造・スキーマ規約・ページフォーマッツ・ツールはすべて、
ドメイン・好み・使う LLM によって変わる。すべてが任意でモジュール式。
使えば便利で、使わなければ不要。
